# 已实现功能与 TODO 范围说明

## 当前实现范围
当前分支交付的是 `中期检查核心 MVP + 角色化工作区版本`。  
目标是让系统可演示、符合课程要求，并且方便其他组员继续扩展。

## 已实现功能

### 1. 登录与角色权限
- 本地登录页面
- 本地账号数据：`data/accounts.json`
- 基于 `HttpSession` 的登录状态保存
- 按角色跳转不同工作区
- 未登录访问业务页时自动重定向回 `/login`
- 已登录用户只能访问自己角色允许的页面
- 登出功能

### 2. TA 流程
- TA 仪表盘
- TA 资料创建与更新
- 本地 CV 上传
- CV 文件类型校验
- 岗位列表，只显示开放岗位
- 岗位详情页面
- 提交申请
- 申请优先级选择
- 我的申请状态页面
- 拒绝状态对 TA 可见

### 3. MO 流程
- MO 仪表盘
- 创建岗位
- 编辑岗位
- 将岗位保存到本地 JSON 数据
- 关闭岗位
- 申请人审核页面
- 将申请标记为 `Under Review`
- 将申请标记为 `Accepted`
- 将申请标记为 `Rejected`

### 4. Admin 流程
- 工作量仪表盘
- 已录取课程数量汇总
- 每位 TA 的已录取课程列表
- 超负荷风险标记
- Admin 作为全局监控角色查看 workload，而不是直接参与录取

### 5. 当前 UI / 交互
- 统一的蓝色 + 品红色视觉方向
- 左侧角色导航
- 顶部导航栏和用户信息区
- 页面右下角滚动按钮
- AI 页面中的可选功能默认折叠显示
- 全局搜索（Search）功能，可搜索岗位、申请和 TA 资料
- 收件箱（Inbox）功能，用于状态更新和系统通知

## 已实现的核心规则
- 不使用数据库
- 所有结构化数据存储为 JSON 文件
- CV 文件保存在本地目录
- 一名 TA 最多申请 `3` 门课程
- 一名 TA 最多被录取 `3` 门课程
- 已关闭岗位不能继续申请
- 被拒绝的申请不会消失，TA 仍可看到
- Admin 页面根据已录取申请计算 workload
- AI 当前不进入核心验收要求

## 组员继续开发时应避免破坏的点
- 最多申请 3 门的规则
- 最多录取 3 门的规则
- 关闭岗位后禁止申请的逻辑
- 本地 JSON 存储方案
- 当前已经使用的页面路由命名
- 登录后的角色跳转与权限控制

## 安全扩展建议
如果其他组员要继续在这个版本上开发，建议顺序是：
1. 先保持现有核心流程稳定
2. 补测试
3. 补 UI 与交互优化
4. 再完成 Search / Inbox
5. 最后再加 AI 增强功能

## 推荐分工方向
- 一名组员：Search / Inbox
- 一名组员：AI 匹配与评分原型
- 一名组员：表单校验与 UI 优化
- 一名组员：测试与 demo 脚本
- 一名组员：README 与用户手册整理

这样可以在不影响当前 MVP 的情况下，继续让每个人都有明确贡献空间。
